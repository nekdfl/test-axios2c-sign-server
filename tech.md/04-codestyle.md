# 4. Стиль кода и соглашения

## POSIX и C

- В `**src/demosign.c**` и `**src/demoposix.c**` задаётся `**_POSIX_C_SOURCE 200809L**`: только стандартная библиотека и POSIX (без Win32).
- В `**src/main.c**` то же макроопределение для единообразия; сигналы через `**sigaction**` (обёртка `DemoPosix_*`), не через устаревший `**signal**` для критичных мест.

## «WinAPI-стиль» в публичном API

Заголовки `**demosign.h**` / `**demoposix.h**` используют:

- типы `**BOOL**`, `**DWORD**`, `**LPCSTR**`;
- пустые макросы `**WINAPI**` / `**CALLBACK**` (на Unix не меняют вызов);
- имена функций `**Component_Action**` (`DemoSign_ComputeDocumentSignature`, `DemoPosix_Sigaction`).

Реализация при этом остаётся обычным C: не тащите в `**demosign.c**` заголовки Axis2.

## Axis2 / сервис

- Следуйте паттерну **skeleton** из документации Axis2: `axis2_get_instance`, `invoke`, `on_fault`, `free`, `init` — порядок в структуре зависит от версии Axis2; у нас инициализация через **именованные поля** в `{ .invoke = ..., .init = ... }`, чтобы не ломаться между версиями.
- Ошибки в окружении Axis2: `AXIS2_ERROR_SET`, сообщение через `axutil_error_set_error_message` (сигнатуры Axis2 могут требовать не-`const` — каст оформляйте явно и узко).

## Форматирование

- `**AM_CFLAGS`**: `-std=c11 -O2 -Wall -Wextra -Wpedantic` в модулях, где задано.
- Не раздувайте дифф: правки по делу, без массового переформатирования чужого кода (в т.ч. лицензионного `main.c` от Axis2 — минимальные вкрапления).
